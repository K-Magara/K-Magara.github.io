Zumo Zumo32U4 BLEserial3 Bluetooth BLE Python Wifi Camera

マリオカートライブもどき試作３号機

はじめに

　試作2号機でもまずまずの結果が得られましたが，2つのESP32を使うのはコスト的によくないということで， 新たに M5camera を用意して制御系の1チップ化を図ります． M5camera は PSRAM 搭載のため，これまでのカメラよりも高性能化が見込めるはずです． 親切に Type-C のケーブルまで付属． 試作2号機製作時に購入した Typc-C の変換アダプタが無駄に... 付属品は短いので購入したケーブルを使いますけどね．
　Zumo32U4 との 通信用に，GROVE端子から信号を取り出す必要があるので， GROVE - 4ピン-ジャンパメスケーブル (5本セット) も併せて購入しました．

試作完成品

　先に完成した画像を以下に示します． M5camera に標準で付属していた LEGO の固定パーツを活用するため，ちょこっと細工しました．

以下にハマった点を列挙します．

NimBLEがビルドできない

　PSRAM搭載カメラなので，ESP32用のボードライブラリを2.01にアップグレードしました．詳細は試作1号機のページを参照． ところが今度はNimBLE（1.33）を使ったスケッチがビルドできなくなる症状が発生．CPUが高負荷のままArduinoIDEが落ちてしまいます． やむなくNimBLEの使用はあきらめ，ESP32標準のBLEライブラリに変更しました．ひな型は ESP32 BLE Arduino → BLE_uart をベースにしています．
　後日談ですが，編集中にESP32用のボードライブラリが2.02へ，NimBLEも1.34にアップデートされていたので，いまならビルドできるかもしれません．

BLEで通信できない

　これはプチハマリでした．PythonからM5cameraのBLEが見えないと怒られますが，よくみるとMacアドレスが違いました． スケッチ書き込み時のMacアドレスは，24:62:ab:d9:dd:b0 なのに

BLE用のMacアドレスが 24:62:ab:d9:dd:b2 というオチでした．

ソフトウェアシリアルが使えない

　BLEの単体テストでは，GORVE端子のGPIO4と13をソフトウェアシリアルとして設定し，Zumo32U4 への指示を出していましたが， WiFiカメラとの組み合わせでソフトウェアシリアルが使えないことが判明しました． 


	SoftwareSerial	mySerial(4, 13);	// WiFiカメラと併用するとESP32がリブートを繰り返す
どうしようもないのでソフトウェアシリアルをあきらめ，本来の用途である I2C での通信に変更しました．

I2Cで通信できない

　これがドハマりでした．というよりI2C通信の仕組みを理解していなかったのが原因． シリアル通信と同じように，垂れ流したデータを適当に受信すれば通信できるようなイメージでしたが，全く受信できない． あげくに Wire.endTransmission() の戻り値が，ヘルプには 0～4 の定義しかないのに 5 を返します．エラーコードの調査のしようがなく途方に暮れておりました．
　かなり悩みましたが，結局 "受信の割り込み" を定義することで正しく受信できるようになりました． やみくもに available() してもマスターとのタイミングが合わないと読めないことがわかりました． 以下は Zumo32U4 （スレーブ）側の抜粋です． 


#define I2C_ADDRESS   0x10
volatile byte var = 0;

void dataRecv(int)
{
  if ( Wire.available() ) {
    var = Wire.read();
    ledRed(1);
  }
}

void setup()
{
  Wire.begin(I2C_ADDRESS);  // slave
  Wire.onReceive(dataRecv); // ISR
  Serial.begin(9600);
  while(!Serial);
}

void loop()
{
  Serial.println(var, HEX);
  ledRed(0);
  delay(1000);
}
